2What Already Works Well

Assignment provenance is excellent. aglet show explains every assignment in plain language ("Inherited from child Sheffield Financial", "Auto-classified by implicit_string: matched also-match term '(maybe)'"). This is the single best trust-builder in the product — several recommendations below amount to "surface this existing provenance earlier."
Undo is real and communicative. d → Ctrl-Z reported "Undid mark done" and restored the item; the hint row contextually revealed redo. (Note: docs/specs/product/gaps.md still claims undo doesn't exist — doc drift.)
Recovery story exists. Deletion log + restore with actor attribution (by=user:tui) is a solid safety net.
Error messages with agency. Ambiguous prefix errors list all matching UUIDs; unconfigured workflow says exactly where to configure it. Exit codes and stderr usage are correct.
The View Editor is genuinely good. Settings/Sections/Details/live-Preview layout reads clearly; the preview showing per-section match counts is a great feedback loop. Esc/dirty-confirm semantics are consistent.
Empty states teach. "No items. n:add item v:switch view q:quit" — exactly right.

3Priority 1 — Trust & Data-Integrity Pain Points

Cases where the system changes or ignores user data without telling them. These erode trust in an app whose core promise is "your items go where they should."

P1TUI · When editingLenient date parsing silently ignores part of what the user typed

Observed: In the edit panel, appending garbage to the When field (2026-06-12 00:00X, or …next wek) and pressing Ctrl-S saved and closed with no message. The parser (board.rs:5157-5189) falls through to BasicDateParser, which extracts any recognizable date token from within the string and discards the rest. The user cannot tell whether their edit took effect; a typo like "next wek" appended to an existing date is silently dropped.

On save, when the parsed result differs from the literal input, echo the normalization: When: saved as 2026-06-12 (interpreted from "2026-06-12 00:00X"). Better: live parse-preview line under the When field (the inline When popup already does pane-local feedback — reuse that pattern in the full edit panel), and reject input with unconsumed trailing tokens.

P1TUI · Add itemSection auto-assign categories are invisible at the moment they fire

Observed: Adding an item to the TODOs section showed only Adding to "TODOs" (auto-assign 1 categories) — without naming the category. The section was (mis)configured with Auto-assign on add: Overdue, so every new TODO instantly displayed Priority "Overdue" with no explanation. The misconfiguration was only discoverable four levels deep in the View Editor's section details. (Also: "1 categories" — grammar.)

Name the categories in the context line: Adding to "TODOs" — will assign: Overdue. Consider a View Editor lint that warns when an auto-assign category is not part of the section's criteria (the natural configuration is for them to match).

P1Core · Implicit matchingNote text triggers semantic-feeling misclassifications

Observed live: A bill item's note "Paid via app, maybe only works during biz hours?" auto-assigned the GTD category Someday/Maybe via also-match term "(maybe)" — silently re-filing a dated payment into the someday bucket. AGENTS.md already documents this class of surprise (notes matching Ready/CLI/TUI).

Three dials, any of which would help: (a) default also-match/implicit matching to title only, with note matching opt-in per category; (b) route note-text matches through the suggestion queue (Suggest/Review) instead of auto-apply, reusing the existing literal/semantic split; (c) show a transient toast when a save triggers auto-assignments: Auto-assigned: Someday/Maybe (matched "maybe" in note) — u to undo. The provenance already exists; it's just never pushed to the user at assignment time.

P1TUI · DeleteDelete confirmation is a footer-only prompt that doesn't name the target

Observed: x shows "Delete item? y:confirm Esc:cancel" in the bottom status row only — two hundred columns away from the selected item, with no item title, and visually identical to any other status line. The same mode is multiplexed with the done-blocker-cleanup prompt. Deletion is recoverable only via CLI (aglet deleted / restore); the TUI has no restore surface.

Use a small centered confirm popup naming the item ("Delete 'Wash DRZ'? This is logged and restorable via CLI."), or at minimum include the truncated title in the footer prompt. Longer term: a TUI deleted-items browser (the data model already supports it), and include delete in Ctrl-Z undo if it isn't.

4Priority 2 — Discoverability & Consistency

Aglet inherits Lotus Agenda's flexibility-overwhelm problem (the Wikipedia quote in your own README). Most of these findings are about the gap between the app's actual coherence and what the chrome communicates.

P2TUI · HelpThe help panel clips its bottom third and breaks alignment

Observed at 220×50: the GLOBAL section (settings, category manager, undo, quit!) is cut off with no scroll affordance — render_help_panel renders a fixed Paragraph with no scroll state (render/mod.rs:4830). Key-column padding breaks for wide keys: "v/V/F8 ,/. gaViews, previous/next view", "S-↑/S-↓Move item", and unicode arrows misalign rows.

Make the panel scrollable (j/k/PgDn like the inspector popup), widen the key gutter to fit the longest key string, and consider two columns at wide terminal sizes — the content easily fits 220 cols side-by-side.

P2TUI · FooterThe two footer rows contradict each other and silently drop hints

Observed: View palette status row says "n new, d datebook, r rename, x delete, e edit"; the hint row below says "N:new c:clone r:rename e:edit x:delete" — different case for new, clone vs datebook each appearing in only one row. Category manager: status row says "H/J/K/L move, <</>> shift level", hint row says "S-↑/↓:move H/L:level". The Normal-mode hint row carries ~20 entries and entries appear/disappear as state changes (redo appeared, settings hints fell off), so users can't build spatial memory. Datebook keys {/}/0 never appear in the footer at all, even in a datebook view.

Single source of truth: generate both rows from one keymap table per mode so they can't diverge. Curate Normal mode to ≤10 stable, high-frequency hints + "?:more". Make hints context-sensitive by view type (datebook keys when in a datebook).

P2TUI · PickersPopup text inputs render in the global footer, far from the popup

Observed: In the assign picker, typing / puts the filter text ("Category> moto") in the bottom screen footer while the filtered results live in a centered popup — 20+ rows of eye travel, and the typed query is easy to miss entirely. AGENTS.md notes the same family of issues with Category Manager inline-edit cursors.

Render the input row inside the popup (the Link Wizard already does this correctly with its "Search>" box — make it the standard).

P2TUI · Assign pickerThe dual-marker checkbox language is cryptic

Observed: Rows render like [-] [x] TODO [exclusive] with legend "[+] add [-] remove [x] assigned [ ] not assigned". Two adjacent brackets (pending-change + current-state) require the legend to decode, and a [-] appeared on TODO before any user toggle (derived/subsumption state interacting with pending markers). Exclusive-group radio behavior (toggling one sibling clears another) is not signaled before it happens.

One marker per row with color/word for the delta: [x] TODO, [x→ ] TODO (will remove), [ →x] Home (will add); dim derived-only assignments and label them ("via Sheffield Financial") using the provenance that show already has. Show "(replaces: Low)" on exclusive siblings before apply.

P2TUI · Category treeFiltered trees keep indentation but hide ancestors

Observed: Filtering categories on "moto" rendered "Moto Expenses 2026" and "Adams Motorsports Park" at their original depths with all parents (Finance → Expenses…) hidden — indentation with nothing to be relative to. With 60+ categories in a real DB this makes filter results genuinely ambiguous (two "2025"/"2026" leaves exist under different parents).

Either show matched nodes flat with a breadcrumb suffix ("Moto Expenses 2026 — Finance ▸ Expenses") or keep dimmed ancestor rows in the filtered tree.

P2TUI · SearchEnter's search-or-create duality risks accidental creation

Observed: The persistent bar reads "Search or create…"; in section search, Enter is "jump/create"; in the assign-picker input, Enter creates a new category when there's no exact/single match. The precedence rules (exact match → single visible match → create) are sensible but invisible, so a typo'd search plus Enter mints a new item/category. Also surprising: Enter on a global-search hit opened the full edit panel rather than just navigating to the item.

Make creation explicit at the moment of commit: when Enter would create, require one more keystroke with a labeled prompt ("No match — Enter again to create item 'reserach: flights'"), or reserve create for Ctrl-Enter. Have global-search Enter select/reveal, with e to edit.

P2TUI · Edit vs AddAdd and Edit panels present different category models

Observed: Add Item shows an inline Categories checklist pane; Edit Item shows an "Actions" pane (a Assign categories, i Inspect) — categories are a hop away behind the full picker. Same item, same concept, two layouts and two muscle memories.

Converge on the inline checklist in both panels (Edit's checkboxes already initialize from full assignment state per AGENTS.md), keeping a as a shortcut to the full picker for deep trees.

P2TUI · Board headerSection headers conflate title, count, and column labels

Observed: The first section of "TODO list" renders as (4)─────When ─── TODOs ─── Priority ─── Area — a bare count, then column headers, with the section's identity carried only by the item-column label. The third section's item column is labeled "Done", which collides with the reserved Done concept and reads as a status column. Meanwhile "Overdue (1)" renders title-first. Which text is the section name is guesswork.

Always render Title (count) as the leading element of the divider, visually distinct (bold/color) from column headers; never let the item-column label visually replace the section title.

P2TUI · DatebookPeriod browsing jumps a year, and recurrences vanish from future buckets

Observed: In a month-bucketed datebook, } jumped June 2026 → June 2027 (window-length steps), though README says "next period" and the sections are months. A monthly recurring bill (DRZ Payment ↻) appears only in its next-occurrence bucket; July onward show (0) — so the natural "upcoming bills" reading of a datebook is wrong.

Step by one bucket on {/} (window-jump on Shift variants), and project recurrence occurrences into future buckets as ghost entries (dimmed, "↻ projected") without materializing items.

P2TUI · ClassificationProvider misconfiguration surfaces as per-save noise instead of a settings warning

Observed: Every save flashes "Classification complete: no new suggestions (OpenAI: OPENAI_API_KEY not set)". Global Settings shows provider OpenAI / gpt-5.4-nano with no indication the key is missing. The one place the user could fix it is the one place that doesn't mention it.

Validate the provider in Global Settings (inline "⚠ OPENAI_API_KEY not set" next to the provider row); after the first failed save-classification, suppress repeats for the session. Also: the category manager's top status line ("Literal: Auto-apply | Semantic: Suggest/Review | Ready queue: (unset) | Claim result: (unset)") is jargon-dense — hide unset workflow fields unless configured.

5Priority 3 — Polish & Output Quality

P3BothRaw machine formats leak into user-facing surfaces

Preview/Info and CLI show When: 2026-06-13 00:00:00 (midnight noise on date-only items), Created: 2026-05-09T17:53:53.026833Z (microseconds, UTC) — while the board already renders dates humanely. Deleted-log lines are raw pipe-delimited. A dangling view alias renders as (deleted:ab362776-…) => Category.

One shared human date formatter (date-only when time is midnight; local time; "3 days ago" optional); prune or label dangling aliases ("(deleted category)") and clean them on view save.

P3BothCategory lists everywhere include the full ancestor/reserved closure

A single bill item lists 14 categories ("Area, Cost, Cost Category, DRZ4SM, Expenses, Finance, … TODO, Vendor, When") in preview, CLI list, and board "All Categories" columns — because subsumed parents and reserved categories (When) are flattened in. The signal (DRZ4SM, Payment, Sheffield Financial) drowns.

Default display to leaf/direct assignments, with parents available behind a toggle or dimmed; exclude reserved plumbing categories (When, Entry) from display lists by default.

P3TUILink Wizard ships an empty mystery box

The wizard's top region renders as an empty bordered rectangle (presumably the source-item display) above Relationship/Target/Matches. Matches rows are prefixed "open | …" — status-as-prefix reads like debug output.

Fill the header with "Source: <item title>" (and count for multi-select); render done/open as a glyph not a pipe-prefixed word.

P3DocsSpec/docs drift misleads both users and agents

gaps.md says undo is empty/no-undo-contract (it exists and works); the TUI redesign spec describes save keys and regions that differ from shipped code; older proposals describe pre-live sticky semantics; AGENTS.md itself flags all of these. The README cheat-sheet omits newer keys that exist in-app (f/F numeric format, s sort, m/z lanes, g s settings).

One pass to retire/update stale specs (your own AGENTS.md lifecycle rules cover this), and generate the README cheat-sheet from the same keymap table proposed for the footer.

6CLI-Specific Findings

P2CLI · view showBoard columns are declared but never rendered — finance views are TUI-only

Observed: view show "Moto Budget Combined" prints columns: When [when,w=12] | Vendor [standard,w=26] | Cost [standard,w=12], then a generic ID/STATUS/WHEN/TITLE table with no Vendor/Cost values and no section sums. The view's entire purpose (cost tracking) is invisible from the CLI; the columns header is a tease. AGENTS.md confirms column rendering is TUI-only.

Render configured columns as table columns in view show (and totals rows where summaries are configured); JSON format should include column values too. This is the single highest-leverage CLI improvement for agent workflows.

P2CLI · listDefault list output is built for parsing, not reading — and serves neither perfectly

Three lines per item, full 36-char UUIDs, raw ISO whens, exhaustive category closures, and a STATUS column that says open even when the item's workflow Status category is "In Progress" (documented gotcha). Short-prefix matching exists on input but outputs never exploit it.

Default to a compact one-line row: 8-char id prefix, humane date, title, leaf categories; --verbose for today's output. Either populate STATUS from the workflow Status family or rename the column (done?) to stop implying it.

P3CLI · workflowClaim workflow is CLI-operated but TUI-configured

aglet ready errors with "set Ready Queue and Claim Target categories in TUI Global Settings" — a good message, but the primary consumers of claim are headless agents that may never open the TUI.

Add aglet workflow set --ready <cat> --claim <cat> (and workflow show); keep the TUI settings as the same storage.

P3CLI · ergonomicsMulti-step authoring requires N round-trips

The canonical create flow is add + one category assign invocation per category (AGENTS.md documents the awk-extraction pattern). Each is a separate process launch; agents and humans both pay it.

add "title" --category Work --category High --when friday (repeatable flag, already idiomatic with list) would collapse the pattern; print assignment results in the same output.

7Recommendation Summary

Quick wins (small diffs, high trust payoff)

  • Name auto-assign categories in the add-item context line; fix "1 categories".
  • Echo the normalized When after save when it differs from typed input; show parse feedback in the edit panel like the inline When popup already does.
  • Put the item title in the delete confirmation; mention restorability.
  • Make the help panel scrollable and fix its key-column alignment.
  • Inline "key not set" warning next to the semantic provider in Global Settings; stop repeating it on every save.
  • Show {/}/0 hints when a datebook view is active.
  • Label dangling aliases as deleted categories instead of printing raw UUIDs.

Structural (one design, many symptoms)

  • Single keymap table per mode driving footer status row, hint row, help panel, and README cheat-sheet — eliminates every contradiction found in §4 at once.
  • Provenance-forward assignment UX: toast on auto-assignment, derived-vs-manual distinction in pickers and category lists, leaf-first category display. The data already exists; this is presentation.
  • Explicit-create commit step everywhere Enter can currently mint a new item/category from a search box.
  • CLI view-column rendering + compact one-line list default, so the CLI tells the same story as the TUI.

Worth a design discussion before touching

  • Implicit matching scope (title vs note; auto-apply vs suggest) — changes classification behavior, the heart of the product. Recommend suggest-not-apply for note-only matches as the conservative first step.
  • Datebook recurrence projection — display-layer ghosts vs materialized occurrences interacts with the recurrence engine and series carry semantics.
  • Add/Edit panel convergence — touches the InputPanel architecture mid-evolution (Phase 5 just completed).
